# Decisions ‚Äî Cute Baby Meditation

Decisions resolved during design discussion and the grill interview. One entry per decision.

## Baseline (pre-grill discussion)

- **Stack: reuse HRV wholesale** ‚Äî React 19, TypeScript strict, Vite 8, Tailwind 4, PWA (Workbox), Vitest, ESLint, multi-version GitHub Pages deploy, Pake desktop shell. Reference clone at `.reference/hrv` (gitignored). *Rationale: all tooling decisions already made and proven in HRV.*
- **Single practice, two app states: Practice / Collection** ‚Äî reuse HRV's switcher UI for the two modes. *Rationale: app has one practice but needs a media-management surface.*
- **Library named "Collection"** ‚Äî not "Cuties". *Rationale: safer tone.*
- **Media in IndexedDB, settings/prefs in namespaced localStorage** ‚Äî plus `navigator.storage.persist()` and an export/import backup. *Rationale: localStorage 5MB can't hold media; blobs store natively in IndexedDB; eviction is the worst failure mode.*
- **Import via file picker, drag-drop, clipboard paste** ‚Äî social-link extraction out of scope. *Rationale: link extraction needs a backend/scraping and violates the no-backend privacy stance; paste covers much of the need.*
- **Images re-encoded on import** ‚Äî decode ‚Üí downscale (~2000px max) ‚Üí WebP/JPEG. Undecodable files rejected loudly. *Rationale: HEIC compatibility outside Safari; ~10x storage savings.*
- **Photos and videos, local files only.**
- **Tags, not categories** ‚Äî seeded defaults, user-created allowed. Session setup filters by tags. Named presets deferred. *Rationale: tags are strictly more expressive; presets fall out naturally later.*
- **Hold = one granular event** ‚Äî source, duration, timestamp; multiple holds on one source are separate entries, never summed/averaged at write time. *Rationale: raw events keep every future metric derivable.*
- **Release keeps the current source** ‚Äî re-hold allowed; swipe left = next random, swipe right = back through session history. *Rationale: matches the book (gaze at the same picture again); preserves user agency.*
- **Metric named "aww factor"** ‚Äî formula deferred; derived from raw events.
- **i18n EN/PT-BR kept** ‚Äî HRV strings infra reused.
- **No app-generated audio at all** ‚Äî no ambient, no end-of-session chime, no `src/audio/` port. Videos carry the only sound; single toggle: video sound on/muted (default on). Background music = user's own streaming app; About copy notes unmuted videos interrupt it (iOS audio session). *Rationale: any app audio would pause the user's background music on iOS anyway; drops the largest HRV subsystem; session end is visually unmissable.*
- **Unmuted video playback is gesture-driven** ‚Äî session start tap + swipes grant user activation; one persistent `<video>` element unlocked at Start for iOS.
- **Privacy is the headline feature** ‚Äî user's own photos never leave the device; README leads with it.

## Grill interview

- **Q: Where do session records and granular hold events live?** ‚Üí **IndexedDB**, alongside media. Only settings/prefs/UI state in localStorage. *Rationale: events grow unboundedly (~1MB/yr for daily use); localStorage's 5MB origin-shared cap would eventually silently truncate; one export path covers media + history together.*
- **Q: Timer expires mid-hold?** ‚Üí **Wait for release.** Session enters overtime, the hold keeps recording; completion screen appears on release. Overtime stored in the session record. *Rationale: never interrupt the feeling ‚Äî that's the practice; the data stays honest.*
- **Q: Duration control?** ‚Üí **HRV-style stepper, 1‚Äì30 min, default 5 min** (the book's prescription), persisted as last-used. *Rationale: familiar HRV idiom, direct port.*
- **Q: Stop/pause semantics?** ‚Üí **Stop only, no pause.** Stop confirms (HRV-style), ends session; all recorded hold events kept; session saved with endReason `stopped` vs `completed`. *Rationale: releasing a hold already costs nothing, pause adds a state and a control for no practice benefit; granular data is never discarded.*
- **Q: Gesture grammar on the session surface?** ‚Üí **Move past threshold = swipe; release <~300ms = tap (toggles overlay visibility, not recorded); ‚â•300ms = hold, recorded in full including the first 300ms.** *Rationale: clean disambiguation, no accidental-tap pollution in the data, and the discrete overlay can vanish entirely video-player-style.*
- **Q: Keyboard controls?** ‚Üí **Full mapping: Space held = hold, ‚Üê/‚Üí = back/next, Esc = stop (confirmed), short click/O = toggle overlay.** Key-repeat filtered on Space. *Rationale: cheap to build alongside pointer gestures, hard to retrofit; keyboard-only accessibility.*
- **Q: Random selection policy?** ‚Üí **Shuffle-bag:** shuffle filtered pool at start, deal in order, reshuffle on exhaustion (no back-to-back repeat at the boundary). Swipe-back history = the dealt sequence. *Rationale: guarantees full coverage before repeats; small pools feel fair.*
- **Q: Video behavior in session?** ‚Üí **Autoplay from start on display, loop until navigation; holds don't affect playback.** *Rationale: matches the reels/shorts habit the sources come from; playback decoupled from the metric gesture.*
- **Q: Fullscreen mechanism?** ‚Üí **CSS full-viewport takeover only** (HRV session style, safe-area insets); no Fullscreen API in v1. *Rationale: works identically everywhere including iPhone Safari; installed PWA/Pake already chromeless; avoids the Esc double-meaning conflict.*
- **Q: Overlay contents?** ‚Üí **Countdown (remaining ‚Üí 0:00, then +overtime), clock, stop button, plus a barely-there hold-in-progress indicator** (thin ring/soft glow). Semi-transparent pill with backdrop blur for any-background contrast; tap hides all of it. *Rationale: user set a duration and wants what's left; first-timers need confirmation the hold registered without attention theft.*
- **Q: Import/tagging flow?** ‚Üí **Import first, tag after.** Files land untagged instantly; Collection grid multi-select ‚Üí assign tags; untagged sources count under "all tags". Seeded tag list: Babies, Kittens, Puppies, Family, Bhakti (user-editable). *Rationale: zero friction on drop/paste/batch; tagging is curation, done at leisure.*
- **Thumbnails (engineering):** ~300px thumbnail blob generated at import, stored alongside the source; Collection grid renders thumbs only. *Rationale: grid must not decode full-size media.*
- **Q: Video import validation?** ‚Üí **Probe + store as-is.** Import loads the file into a `<video>` to verify decodability and captures a poster frame (= thumbnail); undecodable files rejected loudly with a format hint. No size cap; per-item size shown and a total storage gauge via `navigator.storage.estimate()`. *Rationale: no client-side transcode is feasible; unplayable media must fail at import, never mid-session.*
- **Q: Source deletion semantics?** ‚Üí **Tombstone.** Blob + thumbnail removed (space freed); source record kept with deleted flag, caption/type/tags retained; hold events untouched. *Rationale: history and lifetime totals stay truthful; costs ~100 bytes per deleted source.*
- **Q: Backup format/semantics?** ‚Üí **Single .zip (media + metadata/events JSON) via fflate ‚Äî accepted as first runtime dep (~8KB). Import = full restore replacing current state after explicit confirm.** Merge-on-import deferred. *Rationale: zip survives big libraries streaming-wise where base64-JSON dies; replace semantics avoid id-conflict complexity in v1.*
- **Q: Aww factor v1 formula?** ‚Üí **Sort key = lifetime total hold time per source; cards show raw stats ("14 holds ¬∑ 9m"), no synthetic score.** *Rationale: transparent and obviously correct; granular events let the formula evolve later without migration.*
- **Q: Stats page scope?** ‚Üí **Modest: lifetime totals (sessions, practice time, hold time, longest hold) + recent sessions list.** Reuses HRV StatsPage shell; no charts in v1. *Rationale: makes the granular log visible without polish-before-product.*
- **Q: Branding?** ‚Üí **App name "Cute Baby Meditation"; keep HRV's Mono Zen light/dark palette.** Deploys at `lucindo.github.io/cute`; localStorage keys prefixed `cute:`. *Rationale: the book's own name self-explains; neutral chrome recedes behind the user's photos; accent tweakable later.*
- **Q: Backgrounding mid-session?** ‚Üí **Wall-clock timer keeps counting while hidden; active hold ends at visibility loss (recorded to that moment); on return, same source or completion screen if expired.** *Rationale: honest timekeeping, no phantom holds; touch state is untrustworthy across visibility changes.*

## Resolved without grilling (obvious given prior decisions)

- **Empty state:** Start is blocked when the tag filter matches zero sources; friendly guidance points to the Collection (echoing the book: "download five to ten pictures"). Minimum pool = 1, no nagging. No bundled starter content ‚Äî privacy stance plus copyright.
- **Learn/About page:** adapted HRV LearnPage ‚Äî practice explanation from the book, the three YouTube links, book credit, and the note that unmuted videos interrupt background music.
- **Wake lock:** kept as in HRV (progressive enhancement) ‚Äî screen must not sleep between holds.
- **Deploy/desktop:** HRV's `deploy.yml` (multi-version Pages + versions.json) and `desktop.yml` (Pake wrapping the live URL) reused with name/URL swapped, per original scope.
- **Deferred beyond v1:** named presets, merge-on-import backup, custom background track, recency-weighted aww formula, charts, Android PWA share-target, Fullscreen API enhancement, optional end chime.
- **Envelope storage ported without HRV's migration ladder.** `readEnvelope` keeps the forward-compat spread and `writeEnvelope` keeps the cross-tab downgrade guard, but `migrateEnvelope` is omitted until the first real schema change (STATE_VERSION starts at 1, no legacy data exists). *Rationale: the guard is live safety; the ladder is dead code until there's something to migrate.*

## Implementation (storage & import build-out)

- **`holdEvents` ships without secondary indexes.** Stats and aww-factor read the full event set anyway; a `bySource`/`bySession` index would be speculative. Adding one later is a routine DB version bump. *Rationale: no consumer issues per-source queries; full scans of years of small event records are milliseconds.*
- **Only *animated* GIF/WebP pass through as-is; static ones re-encode like any still.** Animation is detected by byte inspection at import (GIF: block walk counting image descriptors; WebP: VP8X animation flag). *Rationale: canvas re-encode would freeze animations to frame 1, while static GIF/WebP still gain the WebP re-encode savings.*
- **All IndexedDB writes go through `writeMany` ‚Äî an all-or-nothing multi-store batch.** Import (source+blob+thumb) and tombstoning (source update + blob/thumb delete) commit in one transaction or not at all. *Rationale: a half-applied import or delete is the worst corruption class; one primitive makes atomicity the default.*
- **Storage gauge usage is summed from our own records (live source bytes + thumb sizes); `navigator.storage.estimate()` supplies only the quota.** Deviates from FR-19's letter, which named `estimate()` for usage. *Rationale: the browser's accounting lags IndexedDB blob writes badly ‚Äî Safari reported 59.7 KB for a ~3.7 MB collection; our records are deterministic and update the instant an import or delete lands.*
- **The video probe seeks 0.5 s in for the poster frame and sets no timeout.** Undecodable files surface via the `<video>` `error` event. *Rationale: frame 0 is often black or mid-fade; a blob URL either decodes or errors ‚Äî network-style stalls can't happen.*
- **Media payloads stored as ArrayBuffer + MIME type, not Blob (db v3).** WebKit's IndexedDB fails to store Blob values ("Error preparing Blob/File data to be stored in object store") ‚Äî and on the import's multi-put transaction it hangs without ever firing `abort`, which presented as an eternal "Importing‚Ä¶" on iPhone. Reproduced in Playwright WebKit on both secure and insecure origins; ArrayBuffers persist reliably everywhere, and Blobs are rebuilt on read. Supersedes the baseline "blobs store natively in IndexedDB" rationale. *Cost: file bytes pass through memory at import ‚Äî fine at phone-media scale.* The v3 rung clears pre-release media stores instead of migrating (Blob reads are async and can't run inside a versionchange transaction).
- **Ids come from `newId()` (`src/domain/id.ts`), never `crypto.randomUUID` directly.** `randomUUID` is undefined on insecure origins ‚Äî plain-http LAN testing on a phone ‚Äî which crashed import; `getRandomValues` exists everywhere and backs a hand-rolled UUID v4 fallback.
- **Q: How are tags modeled and stored?** ‚Üí **A `tags` store (db v2) with `TagRecord { id, name: string | null }`; sources and session tag-filters reference tag ids, never names.** Seeded tags have stable ids (`seed:babies` ‚Ä¶) and `name: null`, meaning the display name resolves from `strings.ts` in the current locale (AC-21 holds across language switches); renaming freezes a literal name. Seeding happens inside the v2 upgrade transaction, so a deleted seed never respawns. Tag creation lives in the assign flow only (create = create-and-assign); the manager handles rename/delete. *Rationale: id references make rename a one-record update and localization free; catalog records make unassigned tags and seed deletion first-class.*
- **Q: Where does per-item curation live after touch testing?** ‚Üí **A per-item bottom sheet:** tapping a tile opens `SourceSheet` (larger preview, file size, tag chips, create-and-assign, delete) ‚Äî replacing the per-tile `√ó` and size row; the grid is thumbnails-only. Select mode stays as the bulk multi-select path (FR-15). `ConfirmDialog` and the sheet share the `Sheet` `<dialog>` primitive. *Rationale: user feedback ‚Äî chips-under-toolbar plus a mouse-sized `√ó` don't work on touch; the sheet gives touch-sized targets and a natural home for caption editing.*

## Implementation (collection editing UI)

- **Q: How are per-item caption/tag edits committed?** ‚Üí **Staged draft, persisted on Save.** The item sheet (`SourceSheet`) holds a local draft of caption + tag set; edits hit IndexedDB only on Save, as one atomic `updateSource(id, {caption, tags})`. Closing (button, Esc, or backdrop) with unsaved edits pops a discard confirm. Creating a tag from the sheet writes the catalog record immediately (tags are shared) but stages only the assignment. Supersedes the earlier write-through-per-toggle behavior. *Rationale: user testing ‚Äî silent per-tap writes surprised; explicit Save/Discard is clearer and gives a natural cancel.*
- **Q: Keep the bulk multi-select tagging flow?** ‚Üí **No ‚Äî removed.** The Select toolbar button, select mode, and shared bulk `TagAssignPanel` wiring are gone, along with the `applyToSources`/`createAndAssign` hooks and the `applyTagToSources` storage op. Per-item tagging via the sheet is the only path. *Rationale: multi-select was poor on touch and not worth fixing now; the item sheet already covers tagging. Drops SPEC FR-15.*
- **Q: Where does tag management (rename/delete) live?** ‚Üí **In a bottom sheet (`TagManagerSheet`), not inline.** "Edit tags" opens the manager in the same `Sheet` shell as the item card rather than expanding inline and shoving the grid. *Rationale: consistent sheet UX; inline disclosure pushed content around.*

## UI fidelity

- **Port HRV's primitive vocabulary; don't hand-roll UI.** Cute reuses HRV "wholesale", but the early UI ported only `PageShell` and hand-rolled the rest (bespoke `ModeToggle`, a generic `Sheet`, inline-Tailwind rows). It drifted visibly from HRV ‚Äî weak centering, flat cards, crude rows. Standing rule: build each screen from the HRV analogue's primitives (`TopAppBar`, `SegmentedControl`, `SettingsRow`/`SettingsStepper`, `SectionCard`, `IconButton`, ‚Ä¶) in `.reference/hrv/src`, swapping `--color-breathing-*` tokens for `--color-zen-*`. Legitimate divergence is allowed only where cute has surface HRV lacks (media collection, tags). *Rationale: HRV's look & feel is a consistent system of primitives; reimplementing them ad-hoc yields a different-looking app and re-derives solved problems.*

## Implementation (session setup)

- **Q: How is the session-setup screen laid out?** ‚Üí **Inline controls on the Practice screen, not HRV's `SetupCard` ‚Üí editor-`Sheet` pattern.** The duration stepper (and the coming tag filter + Start) sit directly on the Practice screen, built from ported HRV primitives (`SettingsStepper`, `SegmentedControl`). *Rationale: HRV's summary-card-opens-a-sheet indirection earns its keep with ~5 practice settings; cute has 2, so a card+sheet would cargo-cult the pattern ‚Äî inline is simpler and still HRV-faithful at the primitive level.*

## Implementation (session runtime)

- **Q: How is the running session modeled?** ‚Üí **A pure, time-injected state machine.** `sessionMachine.ts` is a `running | complete` discriminated union with pure transitions that take wall-clock `now` as a parameter ‚Äî the domain never reads a clock. Completion is **tick-driven, not release-driven**: releasing a hold during overtime ends the hold, and the next wall-clock tick finalizes. *Rationale: determinism + testability; one completion path covers timer expiry, hold release, and backgrounding (FR-36/38).*
- **Q: Where do hold-event ids come from?** ‚Üí **Assigned at persistence, not in the domain.** The machine carries holds as `RecordedHold = Omit<HoldEventRecord,'id'>`; `saveCompletedSession` stamps `newId()` per hold in one transaction with the session record. The domain type-only-imports the storage record shapes. *Rationale: keeps the domain pure and persistence-free without duplicating record types.*
- **Q: Why must overlay controls stop pointer propagation?** ‚Üí **The overlay renders inside the gesture-capturing takeover, so a control's press bubbles to the surface and registers as a tap/hold.** The Stop button `stopPropagation`s pointerdown; without it, clicking Stop toggled the overlay off (tap) and unmounted the button before its `onClick` fired. *Rationale: recurring gotcha for any future overlay control (e.g. the coming mute toggle).*
- **Q: Video sound in-session (v1)?** ‚Üí **Deferred ‚Äî muted autoplay + loop for now.** The session renders `<video autoplay loop muted playsinline>`; sound-on default + iOS first-unmute via a single persistent element unlocked at the Start gesture (FR-35) is its own roadmap item. *Rationale: muted autoplay works everywhere without a user-activation dance; unmuted iOS needs the persistent-element work + device testing.*
- **Q: Tag filter persistence + match semantics?** ‚Üí **Session-local (not persisted), OR match.** A source matches if it carries any selected tag; empty selection = all incl. untagged; the selection resets each Practice visit. *Rationale: a filter is a per-session choice, not a durable preference; OR is the intuitive multi-select behavior.*
