# Decisions — Cute Baby Meditation

Decisions resolved during design discussion and the grill interview. One entry per decision.

## Baseline (pre-grill discussion)

- **Stack: reuse HRV wholesale** — React 19, TypeScript strict, Vite 8, Tailwind 4, PWA (Workbox), Vitest, ESLint, multi-version GitHub Pages deploy, Pake desktop shell. Reference clone at `.reference/hrv` (gitignored). *Rationale: all tooling decisions already made and proven in HRV.*
- **Single practice, two app states: Practice / Collection** — reuse HRV's switcher UI for the two modes. *Rationale: app has one practice but needs a media-management surface.*
- **Library named "Collection"** — not "Cuties". *Rationale: safer tone.*
- **Media in IndexedDB, settings/prefs in namespaced localStorage** — plus `navigator.storage.persist()` and an export/import backup. *Rationale: localStorage 5MB can't hold media; blobs store natively in IndexedDB; eviction is the worst failure mode.*
- **Import via file picker, drag-drop, clipboard paste** — social-link extraction out of scope. *Rationale: link extraction needs a backend/scraping and violates the no-backend privacy stance; paste covers much of the need.*
- **Images re-encoded on import** — decode → downscale (~2000px max) → WebP/JPEG. Undecodable files rejected loudly. *Rationale: HEIC compatibility outside Safari; ~10x storage savings.*
- **Photos and videos, local files only.**
- **Tags, not categories** — seeded defaults, user-created allowed. Session setup filters by tags. Named presets deferred. *Rationale: tags are strictly more expressive; presets fall out naturally later.*
- **Hold = one granular event** — source, duration, timestamp; multiple holds on one source are separate entries, never summed/averaged at write time. *Rationale: raw events keep every future metric derivable.*
- **Release keeps the current source** — re-hold allowed; swipe left = next random, swipe right = back through session history. *Rationale: matches the book (gaze at the same picture again); preserves user agency.*
- **Metric named "aww factor"** — formula deferred; derived from raw events.
- **i18n EN/PT-BR kept** — HRV strings infra reused.
- **No app-generated audio at all** — no ambient, no end-of-session chime, no `src/audio/` port. Videos carry the only sound; single toggle: video sound on/muted (default on). Background music = user's own streaming app; About copy notes unmuted videos interrupt it (iOS audio session). *Rationale: any app audio would pause the user's background music on iOS anyway; drops the largest HRV subsystem; session end is visually unmissable.*
- **Unmuted video playback is gesture-driven** — session start tap + swipes grant user activation; one persistent `<video>` element unlocked at Start for iOS.
- **Privacy is the headline feature** — user's own photos never leave the device; README leads with it.

## Grill interview

- **Q: Where do session records and granular hold events live?** → **IndexedDB**, alongside media. Only settings/prefs/UI state in localStorage. *Rationale: events grow unboundedly (~1MB/yr for daily use); localStorage's 5MB origin-shared cap would eventually silently truncate; one export path covers media + history together.*
- **Q: Timer expires mid-hold?** → **Wait for release.** Session enters overtime, the hold keeps recording; completion screen appears on release. Overtime stored in the session record. *Rationale: never interrupt the feeling — that's the practice; the data stays honest.*
- **Q: Duration control?** → **HRV-style stepper, 1–30 min, default 5 min** (the book's prescription), persisted as last-used. *Rationale: familiar HRV idiom, direct port.*
- **Q: Stop/pause semantics?** → **Stop only, no pause.** Stop confirms (HRV-style), ends session; all recorded hold events kept; session saved with endReason `stopped` vs `completed`. *Rationale: releasing a hold already costs nothing, pause adds a state and a control for no practice benefit; granular data is never discarded.*
- **Q: Gesture grammar on the session surface?** → **Move past threshold = swipe; release <~300ms = tap (toggles overlay visibility, not recorded); ≥300ms = hold, recorded in full including the first 300ms.** *Rationale: clean disambiguation, no accidental-tap pollution in the data, and the discrete overlay can vanish entirely video-player-style.*
- **Q: Keyboard controls?** → **Full mapping: Space held = hold, ←/→ = back/next, Esc = stop (confirmed), short click/O = toggle overlay.** Key-repeat filtered on Space. *Rationale: cheap to build alongside pointer gestures, hard to retrofit; keyboard-only accessibility.*
- **Q: Random selection policy?** → **Shuffle-bag:** shuffle filtered pool at start, deal in order, reshuffle on exhaustion (no back-to-back repeat at the boundary). Swipe-back history = the dealt sequence. *Rationale: guarantees full coverage before repeats; small pools feel fair.*
- **Q: Video behavior in session?** → **Autoplay from start on display, loop until navigation; holds don't affect playback.** *Rationale: matches the reels/shorts habit the sources come from; playback decoupled from the metric gesture.*
- **Q: Fullscreen mechanism?** → **CSS full-viewport takeover only** (HRV session style, safe-area insets); no Fullscreen API in v1. *Rationale: works identically everywhere including iPhone Safari; installed PWA/Pake already chromeless; avoids the Esc double-meaning conflict.*
- **Q: Overlay contents?** → **Countdown (remaining → 0:00, then +overtime), clock, stop button, plus a barely-there hold-in-progress indicator** (thin ring/soft glow). Semi-transparent pill with backdrop blur for any-background contrast; tap hides all of it. *Rationale: user set a duration and wants what's left; first-timers need confirmation the hold registered without attention theft.*
- **Q: Import/tagging flow?** → **Import first, tag after.** Files land untagged instantly; Collection grid multi-select → assign tags; untagged sources count under "all tags". Seeded tag list: Babies, Kittens, Puppies, Family, Bhakti (user-editable). *Rationale: zero friction on drop/paste/batch; tagging is curation, done at leisure.*
- **Thumbnails (engineering):** ~300px thumbnail blob generated at import, stored alongside the source; Collection grid renders thumbs only. *Rationale: grid must not decode full-size media.*
- **Q: Video import validation?** → **Probe + store as-is.** Import loads the file into a `<video>` to verify decodability and captures a poster frame (= thumbnail); undecodable files rejected loudly with a format hint. No size cap; per-item size shown and a total storage gauge via `navigator.storage.estimate()`. *Rationale: no client-side transcode is feasible; unplayable media must fail at import, never mid-session.*
- **Q: Source deletion semantics?** → **Tombstone.** Blob + thumbnail removed (space freed); source record kept with deleted flag, caption/type/tags retained; hold events untouched. *Rationale: history and lifetime totals stay truthful; costs ~100 bytes per deleted source.*
- **Q: Backup format/semantics?** → **Single .zip (media + metadata/events JSON) via fflate — accepted as first runtime dep (~8KB). Import = full restore replacing current state after explicit confirm.** Merge-on-import deferred. *Rationale: zip survives big libraries streaming-wise where base64-JSON dies; replace semantics avoid id-conflict complexity in v1.*
- **Q: Aww factor v1 formula?** → **Sort key = lifetime total hold time per source; cards show raw stats ("14 holds · 9m"), no synthetic score.** *Rationale: transparent and obviously correct; granular events let the formula evolve later without migration.*
- **Q: Stats page scope?** → **Modest: lifetime totals (sessions, practice time, hold time, longest hold) + recent sessions list.** Reuses HRV StatsPage shell; no charts in v1. *Rationale: makes the granular log visible without polish-before-product.*
- **Q: Branding?** → **App name "Cute Baby Meditation"; keep HRV's Mono Zen light/dark palette.** Deploys at `lucindo.github.io/cute`; localStorage keys prefixed `cute:`. *Rationale: the book's own name self-explains; neutral chrome recedes behind the user's photos; accent tweakable later.*
- **Q: Backgrounding mid-session?** → **Wall-clock timer keeps counting while hidden; active hold ends at visibility loss (recorded to that moment); on return, same source or completion screen if expired.** *Rationale: honest timekeeping, no phantom holds; touch state is untrustworthy across visibility changes.*

## Resolved without grilling (obvious given prior decisions)

- **Empty state:** Start is blocked when the tag filter matches zero sources; friendly guidance points to the Collection (echoing the book: "download five to ten pictures"). Minimum pool = 1, no nagging. No bundled starter content — privacy stance plus copyright.
- **Learn/About page:** adapted HRV LearnPage — practice explanation from the book, the three YouTube links, book credit, and the note that unmuted videos interrupt background music.
- **Wake lock:** kept as in HRV (progressive enhancement) — screen must not sleep between holds.
- **Deploy/desktop:** HRV's `deploy.yml` (multi-version Pages + versions.json) and `desktop.yml` (Pake wrapping the live URL) reused with name/URL swapped, per original scope.
- **Deferred beyond v1:** named presets, merge-on-import backup, custom background track, recency-weighted aww formula, charts, Android PWA share-target, Fullscreen API enhancement, optional end chime.
- **Envelope storage ported without HRV's migration ladder.** `readEnvelope` keeps the forward-compat spread and `writeEnvelope` keeps the cross-tab downgrade guard, but `migrateEnvelope` is omitted until the first real schema change (STATE_VERSION starts at 1, no legacy data exists). *Rationale: the guard is live safety; the ladder is dead code until there's something to migrate.*

## Implementation (storage & import build-out)

- **`holdEvents` ships without secondary indexes.** Stats and aww-factor read the full event set anyway; a `bySource`/`bySession` index would be speculative. Adding one later is a routine DB version bump. *Rationale: no consumer issues per-source queries; full scans of years of small event records are milliseconds.*
- **Only *animated* GIF/WebP pass through as-is; static ones re-encode like any still.** Animation is detected by byte inspection at import (GIF: block walk counting image descriptors; WebP: VP8X animation flag). *Rationale: canvas re-encode would freeze animations to frame 1, while static GIF/WebP still gain the WebP re-encode savings.*
- **All IndexedDB writes go through `writeMany` — an all-or-nothing multi-store batch.** Import (source+blob+thumb) and tombstoning (source update + blob/thumb delete) commit in one transaction or not at all. *Rationale: a half-applied import or delete is the worst corruption class; one primitive makes atomicity the default.*
- **Storage gauge usage is summed from our own records (live source bytes + thumb sizes); `navigator.storage.estimate()` supplies only the quota.** Deviates from FR-19's letter, which named `estimate()` for usage. *Rationale: the browser's accounting lags IndexedDB blob writes badly — Safari reported 59.7 KB for a ~3.7 MB collection; our records are deterministic and update the instant an import or delete lands.*
- **The video probe seeks 0.5 s in for the poster frame and sets no timeout.** Undecodable files surface via the `<video>` `error` event. *Rationale: frame 0 is often black or mid-fade; a blob URL either decodes or errors — network-style stalls can't happen.*
- **Media payloads stored as ArrayBuffer + MIME type, not Blob (db v3).** WebKit's IndexedDB fails to store Blob values ("Error preparing Blob/File data to be stored in object store") — and on the import's multi-put transaction it hangs without ever firing `abort`, which presented as an eternal "Importing…" on iPhone. Reproduced in Playwright WebKit on both secure and insecure origins; ArrayBuffers persist reliably everywhere, and Blobs are rebuilt on read. Supersedes the baseline "blobs store natively in IndexedDB" rationale. *Cost: file bytes pass through memory at import — fine at phone-media scale.* The v3 rung clears pre-release media stores instead of migrating (Blob reads are async and can't run inside a versionchange transaction).
- **Ids come from `newId()` (`src/domain/id.ts`), never `crypto.randomUUID` directly.** `randomUUID` is undefined on insecure origins — plain-http LAN testing on a phone — which crashed import; `getRandomValues` exists everywhere and backs a hand-rolled UUID v4 fallback.
- **Q: How are tags modeled and stored?** → **A `tags` store (db v2) with `TagRecord { id, name: string | null }`; sources and session tag-filters reference tag ids, never names.** Seeded tags have stable ids (`seed:babies` …) and `name: null`, meaning the display name resolves from `strings.ts` in the current locale (AC-21 holds across language switches); renaming freezes a literal name. Seeding happens inside the v2 upgrade transaction, so a deleted seed never respawns. Tag creation lives in the assign flow only (create = create-and-assign); the manager handles rename/delete. *Rationale: id references make rename a one-record update and localization free; catalog records make unassigned tags and seed deletion first-class.*
- **Q: Where does per-item curation live after touch testing?** → **A per-item bottom sheet:** tapping a tile opens `SourceSheet` (larger preview, file size, tag chips, create-and-assign, delete) — replacing the per-tile `×` and size row; the grid is thumbnails-only. Select mode stays as the bulk multi-select path (FR-15). `ConfirmDialog` and the sheet share the `Sheet` `<dialog>` primitive. *Rationale: user feedback — chips-under-toolbar plus a mouse-sized `×` don't work on touch; the sheet gives touch-sized targets and a natural home for caption editing.*

## Implementation (collection editing UI)

- **Q: How are per-item caption/tag edits committed?** → **Staged draft, persisted on Save.** The item sheet (`SourceSheet`) holds a local draft of caption + tag set; edits hit IndexedDB only on Save, as one atomic `updateSource(id, {caption, tags})`. Closing (button, Esc, or backdrop) with unsaved edits pops a discard confirm. Creating a tag from the sheet writes the catalog record immediately (tags are shared) but stages only the assignment. Supersedes the earlier write-through-per-toggle behavior. *Rationale: user testing — silent per-tap writes surprised; explicit Save/Discard is clearer and gives a natural cancel.*
- **Q: Keep the bulk multi-select tagging flow?** → **No — removed.** The Select toolbar button, select mode, and shared bulk `TagAssignPanel` wiring are gone, along with the `applyToSources`/`createAndAssign` hooks and the `applyTagToSources` storage op. Per-item tagging via the sheet is the only path. *Rationale: multi-select was poor on touch and not worth fixing now; the item sheet already covers tagging. Drops SPEC FR-15.*
- **Q: Where does tag management (rename/delete) live?** → **In a bottom sheet (`TagManagerSheet`), not inline.** "Edit tags" opens the manager in the same `Sheet` shell as the item card rather than expanding inline and shoving the grid. *Rationale: consistent sheet UX; inline disclosure pushed content around.*

## UI fidelity

- **Port HRV's primitive vocabulary; don't hand-roll UI.** Cute reuses HRV "wholesale", but the early UI ported only `PageShell` and hand-rolled the rest (bespoke `ModeToggle`, a generic `Sheet`, inline-Tailwind rows). It drifted visibly from HRV — weak centering, flat cards, crude rows. Standing rule: build each screen from the HRV analogue's primitives (`TopAppBar`, `SegmentedControl`, `SettingsRow`/`SettingsStepper`, `SectionCard`, `IconButton`, …) in `.reference/hrv/src`, swapping `--color-breathing-*` tokens for `--color-zen-*`. Legitimate divergence is allowed only where cute has surface HRV lacks (media collection, tags). *Rationale: HRV's look & feel is a consistent system of primitives; reimplementing them ad-hoc yields a different-looking app and re-derives solved problems.*

## Implementation (session setup)

- **Q: How is the session-setup screen laid out?** → **Inline controls on the Practice screen, not HRV's `SetupCard` → editor-`Sheet` pattern.** The duration stepper (and the coming tag filter + Start) sit directly on the Practice screen, built from ported HRV primitives (`SettingsStepper`, `SegmentedControl`). *Rationale: HRV's summary-card-opens-a-sheet indirection earns its keep with ~5 practice settings; cute has 2, so a card+sheet would cargo-cult the pattern — inline is simpler and still HRV-faithful at the primitive level.*

## Implementation (session runtime)

- **Q: How is the running session modeled?** → **A pure, time-injected state machine.** `sessionMachine.ts` is a `running | complete` discriminated union with pure transitions that take wall-clock `now` as a parameter — the domain never reads a clock. Completion is **tick-driven, not release-driven**: releasing a hold during overtime ends the hold, and the next wall-clock tick finalizes. *Rationale: determinism + testability; one completion path covers timer expiry, hold release, and backgrounding (FR-36/38).*
- **Q: Where do hold-event ids come from?** → **Assigned at persistence, not in the domain.** The machine carries holds as `RecordedHold = Omit<HoldEventRecord,'id'>`; `saveCompletedSession` stamps `newId()` per hold in one transaction with the session record. The domain type-only-imports the storage record shapes. *Rationale: keeps the domain pure and persistence-free without duplicating record types.*
- **Q: Why must overlay controls stop pointer propagation?** → **The overlay renders inside the gesture-capturing takeover, so a control's press bubbles to the surface and registers as a tap/hold.** The Stop button `stopPropagation`s pointerdown; without it, clicking Stop toggled the overlay off (tap) and unmounted the button before its `onClick` fired. *Rationale: recurring gotcha for any future overlay control (e.g. the coming mute toggle).*
- **Q: Video sound in-session (v1)?** → **Deferred — muted autoplay + loop for now.** The session renders `<video autoplay loop muted playsinline>`; sound-on default + iOS first-unmute via a single persistent element unlocked at the Start gesture (FR-35) is its own roadmap item. *Rationale: muted autoplay works everywhere without a user-activation dance; unmuted iOS needs the persistent-element work + device testing.*
- **Q: Tag filter persistence + match semantics?** → **Session-local (not persisted), OR match.** A source matches if it carries any selected tag; empty selection = all incl. untagged; the selection resets each Practice visit. *Rationale: a filter is a per-session choice, not a durable preference; OR is the intuitive multi-select behavior.*

## Implementation (session video sound — FR-35)

- **Q: How is unmuted in-session video playback implemented?** → **One persistent `<video>` at the `App` level, unlocked during the Start gesture.** The element outlives the Practice→Session mount swap; `App.handleStart` calls `v.play()/pause()` inside the Start click to grab the iOS user-activation token; `SessionView` then drives the *same* element imperatively (src / muted / play / show) and hands it back hidden + source-less on unmount. The session gesture surface is a transparent `z-20` layer over the `z-10` video (`pointer-events-none`), so taps still reach the takeover. *Rationale: iOS unmuted playback requires the very element that plays to be user-activated during a gesture, and the gesture fires before `SessionView` mounts — only an App-level element exists at that instant. iOS first-play still needs real-device verification; fallback is priming with a tiny blank clip.*
- **Q: Is video sound session-local or a persisted preference?** → **Persisted app pref (`videoSound`, default on).** The in-session overlay toggle mutates the same pref. *Rationale: SPEC lists video sound among app settings; unlike the tag filter, sound is a durable preference.*

## Implementation (aww factor — FR-17)

- **Q: How are per-source hold stats computed?** → **Read-time aggregation from raw `holdEvents`, never pre-aggregated.** `domain/holdStats.aggregateHoldStats` sums count + duration per `sourceId`; `useCollection` joins `holdCount` + `totalHeldMs` onto each source. *Rationale: the granular-data constraint (FR-28) + AC-8 (deleting a source keeps its events, totals unchanged) require deriving totals from the event log at read time.*
- **Q: Aww-sort persistence and per-card stat a11y?** → **Sort is session-local (component state); the per-card stat line is `aria-hidden`.** Aww sort = a stable reorder of the newest-first base by descending total held time. The `♥ n · m:ss` stat is `aria-hidden` so the tile stays named by its caption. *Rationale: a sort is a per-visit choice (like the tag filter); SR parity for the stat is an optional later follow-up.*

## Implementation (settings surface — planned)

- **Q: Where do Theme/Language/Stats/About live, and how are they reached?** → **A full HRV-style Settings page (minus the audio Feedback section), reached via a gear in the `TopAppBar` leading slot on the shell.** Sections: STATISTICS row → Stats page, THEME (`PickerCardGrid` swatch cards, default `system`), LANGUAGE (`SegmentedControl`), ABOUT (version · sha · date + `Source →` to `github.com/lucindo/cute`). A `view` state (`shell | settings | stats`) in `App` routes; back chevrons return. Port HRV primitives (`SettingsSectionHeader`, `IconButton`, `IconAnchor`, `PickerCardGrid`, gear/chevron icons); reuse existing `SegmentedControl` / `SectionCard` / `SettingsRow`. *Rationale: Theme/Language need a home, and the user requires HRV parity minus the audio-only Feedback section.*
- **Q: How is Stats data derived, and how do Theme/Language propagate?** → **Stats aggregates at read time from `sessions` + `holdEvents` (not HRV's pre-aggregated counters); Theme/Language write prefs then dispatch `cute:prefs-changed`, which `useTheme` / `useLocale` re-read.** *Rationale: granular-data constraint + AC-23 (totals must match the event log exactly) rule out pre-aggregated stats; the event-dispatch sync mirrors the existing locale flow. Theme is net-new: `ThemeId` in `domain/settings`, `prefs.theme`, a `useTheme` apply hook with a gated `system` matchMedia listener, and the `index.html` pre-paint script updated to read the persisted theme.*

## Implementation (settings surface — built)

- **`IconAnchor` dropped from the port.** HRV needs it because its TopAppBar persists across a running session and must keep the Settings/Learn anchors focusable-but-disabled mid-practice. Our session is a separate full-viewport takeover, so the shell gear is never shown during a session — `IconButton` covers both the gear and the back chevron. *Rationale: the disabled-anchor state can never fire here; porting it would be speculative.*
- **`PickerCardGrid` ported trimmed — no `optionLayout`/`stack`, no `disabled`.** Stack layout served HRV's audio CuePicker (absent here); `disabled` served mid-session settings (our settings page is unreachable during a session). *Rationale: neither affordance has a consumer; re-adding either is trivial.*
- **Generic `usePreferenceChoice<K>` writer for theme + locale.** One hook writes the merged envelope and dispatches `cute:prefs-changed`; App-side `useTheme` / `useLocale` filter on `detail.key` and re-read. The single-owner writers (`useVideoSound`, `useSessionDuration`) stay dispatch-less. *Rationale: two picker consumers differ only by key; single-owner prefs need no event.*
- **Theme applies via a `data-theme` attribute resolved from a tri-state pref.** `resolveTheme(theme, systemPrefersDark)` collapses `system` via matchMedia; the `index.html` pre-paint treats any non-`light`/`dark` value (incl. `system`, absent, corrupt) as follow-OS — matching `coerceTheme`'s default — so reload never flashes. *Rationale: pre-paint must agree with the runtime coercer or first paint fights hydrated state.*
- **Language endonyms are a module constant, not localized strings.** `English` / `Português` name themselves regardless of active locale; only the section label is translated. *Rationale: standard language-picker UX; avoids duplicating identical option strings across locales.*
- **Source URL confirmed `https://github.com/lucindo/cute`.** Resolves the prior open question; the repo still has no git remote. *Rationale: user-confirmed.*

## Implementation (backup — FR-20/21/22)

- **Q: What goes in the manifest vs the zip, and how is media MIME recovered on restore?** → **The manifest holds records only (sources incl. tombstones, tags, sessions, holdEvents); media bytes ride as `media/<id>.<ext>` + `thumbs/<id>.<ext>` entries.** Blob MIME is recovered from the source record's `mimeType` (authoritative — the extension is cosmetic there); thumb MIME is recovered from the entry extension, which is lossless because thumbs are only ever `image/webp` or `image/jpeg` (encodeImage). *Rationale: keeps the manifest to the SPEC-stated schema; blob MIME is redundant with the source record, and the 2-value thumb space makes the extension a safe MIME carrier without a fragile mime↔ext map.*
- **Q: How strict is manifest validation?** → **`validateManifest` is a fail-loud trust boundary, not prefs-style coerce-and-fallback.** Any bad record rejects the whole array, non-finite numbers are rejected, and clean records are rebuilt field-by-field (unknown keys dropped). Error channel is a human-facing `string`, not `Error`. *Rationale: restore replaces all data, so a half-understood manifest must abort untouched (FR-22/AC-10) — silently defaulting a field would corrupt the restore.*
- **Q: How is restore made atomic and abort-safe?** → **Validate fully in memory first, then clear all six stores and rewrite them in one `writeMany` transaction (clears queued before puts).** A `clear` op was added to `writeMany` for this (its only consumer). *Rationale: a corrupt zip or invalid manifest returns before any store is touched (AC-10); one transaction means replace-all can never half-apply.*
- **Q: Compression level for zip entries?** → **Media stored uncompressed (`level: 0`); manifest.json default-deflated.** *Rationale: images/videos are already compressed — deflating them wastes CPU for no size gain and keeps zipSync fast.*
- **Q: Sync or async fflate?** → **`zipSync`/`unzipSync` on the main thread.** *Rationale: simplest and testable; fine at phone-media scale. The async fflate API is the follow-up if a very large library janks the UI (see PLAN Watch).*
- **Q: How is the download triggered and tested?** → **An injectable `download` seam; the default `triggerDownload` defers `revokeObjectURL` one tick, and `SettingsScreen` exposes an optional `backupDeps` prop to stub it.** *Rationale: iOS Safari cancels an in-flight download if the object URL is revoked in the same tick as the click; the prop matches the existing "inject browser-only leaves" pattern (importImage/importVideo).*
- **Q: Are prefs/localStorage in the backup?** → **No — excluded.** *Rationale: theme/language/video-sound/duration are device-local settings; AC-9 covers only collection, tags, sessions, events, and stats.*
- **fflate returns `Uint8Array<ArrayBufferLike>`, not a valid `BlobPart`/`ArrayBuffer` under strict lib types.** Copy into a fresh `new Uint8Array(...)` before `new Blob`/`new File`. *Rationale: recurring TS gotcha; the copy is unavoidable anyway since Blob/File copy their parts.*

Not a Settings row (my first proposal): the trailing slot was already empty and is HRV's actual placement. `IconAnchor`'s disabled-mid-session state stays dropped — cute's session is a separate takeover, so a plain `IconButton` suffices. *Rationale: HRV parity; the shell already matched gear-left, so About belongs right.*

The panel is a flat stack: practice explanation → how a session works (with the iOS-audio note) → reference videos → Forrest bio → Forrest resources → footer. *Rationale: the multi-practice machinery has no consumer with one practice.*

The affiliation footer + "inspired by Forrest's teachings" live in `content/lockedCopy.ts` behind a frozen byte-equality `.toBe()` test — never a snapshot. *Rationale: reproducing the book's wording is a rights risk; the lock stops a translation/edit from silently changing the affiliation disclaimer.*

## Implementation (PWA — offline + install, FR)

- **Q: How is offline/install implemented?** → **Ported HRV's `vite-plugin-pwa` setup wholesale** — `strategies: generateSW`, `registerType: autoUpdate`, `injectRegister: auto`. No SW-registration code in `src/` (the plugin injects it at build); nothing added to `main.tsx`. Manifest is inline in the plugin config; `start_url`/`scope` are **omitted on purpose** → they auto-default to the Vite `base`, so each versioned Pages build (`/cute/`, `/cute/vN/`) self-scopes and root/versioned SWs don't collide. *Rationale: HRV parity — cute reuses its stack and the same multi-version deploy plan; hand-rolling the SW would re-derive solved problems.*
- **Q: How were the app icons produced?** → **Rendered from the existing `public/favicon.svg` heart via `rsvg-convert`** into 5 PNGs: `pwa-192/512`, `pwa-maskable-192/512` (mark inset ~28% for the maskable safe zone), `apple-touch-icon` (180). The mark is **nested layered hearts** on dark `#1a1d24` — the heart-mark translation of HRV's concentric breathing-ring icon. *Rationale: HRV's icon motif with cute's mark; reads at small sizes and matches the icon system. The SVG sources live only in the scratchpad — regenerate from `favicon.svg` if the icons ever change.*
- **Q: What does the SW precache, and what's trimmed?** → **`globPatterns: js/css/html/png/svg/woff2`** (no `wasm` — cute ships none); **`globIgnores` drops the 5 non-Latin Inter subsets** (cyrillic/-ext, greek/-ext, vietnamese) since locales are EN + pt-BR (both Latin); `cleanupOutdatedCaches: true`. The dropped subsets still emit to `dist/` but stay out of the SW install. *Rationale: trims the precache with no runtime loss; matches HRV.*
- **Colors: manifest `theme_color`/`background_color` are `#1a1d24` (dark, matches the icon bg); the `index.html` `<meta name="theme-color">` stays `#414957` (browser-tab chrome).** *Rationale: mirrors HRV's split — the installed splash/status bar reads dark like the icon, the browser tab keeps the accent.*

## Implementation (performance pass — NFR-1/NFR-2)

- **Q: Did the 500-source library need optimization to hit the targets?** → **No — measured, and it already meets them.** A dev-only seeder (`src/dev/seed.ts`, installed on `window` behind `import.meta.env.DEV`, dynamic-imported so it tree-shakes out of prod) bulk-writes 500 synthetic sources through the real `writeMany` path — distinct gradient thumbnails, ~1280px webp blobs, seeded tags (⅓ untagged), and ~40% with 1–8 holds owned by real session records. Measured against NFR-1 (≤300ms transition) and NFR-2 (≤100ms interaction): grid mount + scroll, session advances, aww-sort, tag filter, and Stats all pass. *Rationale: the perf pass is measure-first; the render paths cleared the bar, so per Simplicity First we ship no speculative optimization.*
- **Scoped-but-unbuilt optimizations, kept on the shelf** in case a heavier real-world library regresses: Tier 1 — grid `<img loading="lazy" decoding="async">` + session `<img decoding="async">` (defers ~490 offscreen decodes); Tier 2 — reuse one persistent IndexedDB connection in `useSession` instead of open/close per advance, and prefetch the next shuffle-bag source during dwell to hide read+decode; Tier 3 — grid virtualization + lazy per-tile thumb object-URLs. *Rationale: `useCollection` eagerly mints 500 object URLs and the grid renders 500 un-virtualized nodes — fine at this scale, but these are the levers if it isn't later.*
- **The seeder is retained** (dev-only) rather than deleted after the pass. *Rationale: re-measuring after any future render-path change is a one-liner (`cuteSeed(500)`); it costs nothing in prod.*
